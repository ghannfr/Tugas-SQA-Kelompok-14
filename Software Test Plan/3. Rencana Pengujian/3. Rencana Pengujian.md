# 3. Rencana Pengujian (Sistem Pengaduan Masyarakat)

Bagian ini menguraikan skenario pengujian yang akan dilakukan pada aplikasi Sistem Pengaduan Masyarakat.

### 3.1 Skenario Pengujian Black Box
| ID Test | Fitur | Skenario Pengujian | Input Data | Expected Result (Ekspektasi) |
|---|---|---|---|---|
| **BB-01** | Pengaduan | Mengirim pengaduan dengan data valid | Isi laporan: "jalan berlubang di lokasi di depan pasar" | Sistem memunculkan notifikasi "Pengaduan Berhasil Dikirim" |
| **BB-02** | Pengaduan | Mengirim pengaduan dengan isi laporan kosong | Isi laporan: (Kosong) | Sistem menolak input dan menampilkan peringatan "Isi laporan harus diisi" |
| **BB-03** | Laporan | Admin memfilter Laporan Bulanan | Bulan: "April", Tahun: "2026" | Tabel menampilkan data pengaduan pada bulan April 2026 |
| **BB-04** | Tanggapan | Admin memberi tanggapan dengan teks kosong | Laporan ID: 012, Teks: (Kosong) | Sistem menolak input dan menampilkan peringatan "Tanggapan tidak boleh kosong" |

### 3.2 Skenario Pengujian White Box
| ID Test | Modul | Deskripsi Pengujian Internal | Expected Result (Ekspektasi) |
|---|---|---|---|
| **WB-01** | `loginAuth()` | Menguji percabangan logika role. Jika `role == 'admin'`, arahkan ke `/admin`. Jika `role == 'masyarakat'`, arahkan ke `/user`. | Sistem berhasil membedakan session dan mengarahkan ke dashboard yang tepat |
| **WB-02** | `exportReport()` | Menguji algoritma perulangan data saat cetak PDF Laporan Harian | Looping berjalan sesuai jumlah baris array tanpa ada data laporan yang terpotong |

### 3.3 Skenario Pengujian Grey Box
| ID Test | Area Pengujian | Skenario Pengujian | Expected Result (Ekspektasi) |
|---|---|---|---|
| **GB-01** | Keamanan URL | Akses URL `/admin/laporan` menggunakan akun masyarakat | Sistem memblokir akses dan mengembalikan ke halaman 403 Forbidden |
| **GB-02** | Sinkronisasi DB | Admin mengubah status laporan menjadi "Selesai" di UI | Kolom `status` pada tabel `tbl_pengaduan` di database berubah secara real-time |
